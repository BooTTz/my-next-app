# 工贸三方监管平台 — 产品设计方案（增强版）

# 1. 总体说明

## 1.1 背景总述

本项目是用于滨州市博兴县级针对工贸类生产加工型企业统一生产安全检查与监管平台。

## 1.2 主要目的

为实现地区工贸企业的安全检查工作的三方监管，实现从政府发布、服务机构执行、企业接收与整改的三方工作完整流程与相应功能在平台的实现。打通原有线下检查资质计划任务和报告三方查阅和执行的不便捷性，提升信息化执行能力，促进检查团队检查和报告的协作关系。

## 1.3 产品形态

一个满足多方登录的在线管理平台，包含区分三个主要用户类型：监管方、服务方、履行方。

### 1.3.1 三方用户对象及主要功能

| 用户类型 | 主要人员类型 | 在平台内主要工作 |
| --- | --- | --- |
| 监管方 | 地方安全管理部门相关人员 | 发布检查计划、检查任务执行情况验收、履行方整体安全情况关注、检查报告综合分析和查看 |
| 服务方 | 地方安全管理服务机构 | 监管方下发的某企业检查任务的执行检查，录入隐患、关联隐患依据及标准文件、制定整改要求，生成检查报告并评估检查结果 |
| 履行方 | 企业用户 | 对服务方检查出的隐患做整改，配合服务方填写相关检查信息，关注自身企业整体安全情况 |

### 1.3.2 终端形态

- 一个Web端平台，采用Next.js + TypeScript + Tailwind CSS + shadcn/ui技术栈，保证平台整体一致性
- 支持用户个人注册并选择用户类型，登录后根据用户类型进入相应工作台，包含相应功能
- 支持创建团队并邀请其他用户选择团队内担任的用户类型并加入，在团队内创建的检查计划、任务与报告默认仅在团队内可见
- 用户必须加入到一个团队中才能正式使用平台功能，提供一个个人工作台页面以显示其在各个团队中的相关检查业务情况，同时作为未加入某个团队时的默认展示页面

# 2. 设计说明

## 2.1 开发规则

### 技术栈

- **框架**: Next.js + TypeScript（App Router模式）
- **样式**: Tailwind CSS
- **组件库**: shadcn/ui（统一使用，禁止引入其他UI库）
- **图表**: Recharts（用于Dashboard统计图表）
- **表格**: TanStack Table（高级表格功能：排序、筛选、分页）
- **表单**: React Hook Form + Zod（表单管理与校验）
- **状态管理**: Zustand（全局状态管理）
- **文件导出**: jsPDF + xlsx（报告导出功能）

### 包管理

- 使用 **pnpm** 作为包管理器，禁止使用 npm 或 yarn

### 命名规范

- **组件文件**: PascalCase（如 `Button.tsx`, `UserCard.tsx`）
- **组件名称**: PascalCase（如 `Button`, `UserCard`）
- **工具函数/hook**: camelCase（如 `useAuth.ts`, `formatDate.ts`）
- **常量**: UPPER_SNAKE_CASE（如 `API_BASE_URL`）
- **页面路由目录**: kebab-case（如 `inspection-plan`, `hazard-rectification`）

### 组件开发

- 优先使用 shadcn/ui 组件
- 新增组件放置于 `components/` 目录
- 每个组件独立文件，使用默认导出
- 组件需包含 TypeScript 类型定义

### 代码风格

- 使用 ESLint + Prettier 进行代码格式化
- 遵循 TypeScript 严格模式
- 保持代码简洁，添加必要注释
- 禁止 `any` 类型

## 2.2 基础通用功能

### 2.2.1 用户管理

用户账号、注册、登录、用户基本信息、用户类型等完整用户管理功能。

**用户实体字段定义：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string(UUID) | 是 | 用户唯一标识 |
| username | string | 是 | 用户名，唯一 |
| password | string(hash) | 是 | 密码（加密存储） |
| realName | string | 是 | 真实姓名 |
| phone | string | 是 | 手机号（用于登录和通知） |
| email | string | 否 | 邮箱 |
| avatar | string | 否 | 头像URL |
| idCard | string | 否 | 身份证号 |
| certNo | string | 否 | 资质证书编号（服务方人员使用） |

| createdAt | datetime | 是 | 注册时间 |
| lastLoginAt | datetime | 否 | 最后登录时间 |

**注册与登录流程：**

1. 注册：填写手机号 → 获取短信验证码 → 填写基本信息（姓名、密码）→ 注册成功进入个人工作台
2. 登录：手机号 + 密码登录，或手机号 + 验证码登录
3. 登录后若已有团队则进入上次退出的团队工作台，若无团队则停留在个人工作台

### 2.4.3 检查计划管理

主要用户为监管方，同时也是作为监管方默认开启的功能。

**检查计划实体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string(UUID) | 是 | 计划唯一标识 |
| teamId | string(FK) | 是 | 所属团队 |
| planNo | string | 是 | 计划编号（自动生成，格式：JHBX-2025-001） |
| name | string | 是 | 检查计划名称 |
| type | enum | 是 | 检查类型：routine(日常检查) / special(专项检查) / random(双随机检查) / holiday(节假日检查) |
| year | number | 是 | 计划年度 |
| startDate | date | 是 | 计划开始日期 |
| endDate | date | 是 | 计划结束日期 |
| description | string | 否 | 计划说明 |
| scope | string | 否 | 检查范围说明 |
| basis | string | 否 | 检查依据（引用的法规标准） |
| creatorId | string(FK) | 是 | 创建人 |
| status | enum | 是 | 计划状态（见下方状态说明） |
| createdAt | datetime | 是 | 创建时间 |
| updatedAt | datetime | 是 | 更新时间 |

**检查计划状态流转：**

```
draft（草稿）→ published（已发布）→ in_progress（执行中）→ completed（已完成）
                                                          ↘ archived（已归档）
```

- draft：创建后默认状态，可编辑
- published：发布后自动创建对应的检查任务并下发至服务方
- in_progress：有任务开始执行后自动流转
- completed：所有任务完成且验收通过后标记完成
- archived：完成后可归档存放

**页面说明：**

- 列表页：检查计划列表，支持按状态/类型/年度/时间筛选，支持查询和排序
- 新建/编辑页：表单页面，填写计划基本信息，关联选择履行方企业（多选）和服务方（为每个企业分配），设置检查时间
- 详情页：计划基本信息 + 关联的检查任务列表 + 整体进度统计

### 2.4.4 检查任务管理

主要用户为服务方，同时也是作为服务方默认开启的功能。

**检查任务实体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string(UUID) | 是 | 任务唯一标识 |
| teamId | string(FK) | 是 | 所属团队 |
| planId | string(FK) | 是 | 所属检查计划 |
| taskNo | string | 是 | 任务编号（自动生成，格式：RW-2025-001） |
| enterpriseId | string(FK) | 是 | 被检查企业（履行方） |
| inspectorIds | string[](FK) | 是 | 检查人员列表（服务方成员） |
| leadInspectorId | string(FK) | 是 | 检查组长 |
| scheduledDate | date | 是 | 计划检查日期 |
| actualStartDate | datetime | 否 | 实际开始时间 |
| actualEndDate | datetime | 否 | 实际完成时间 |
| checkItems | string | 否 | 检查范围/检查项目说明 |
| status | enum | 是 | 任务状态（见下方状态流转） |
| conclusion | enum | 否 | 检查结论：qualified / basically_qualified / unqualified |
| safetyLevel | enum | 否 | 安全等级评定：A / B / C / D |
| remark | string | 否 | 备注 |
| createdAt | datetime | 是 | 创建时间 |
| updatedAt | datetime | 是 | 更新时间 |

**检查任务状态流转：**

```
created（已创建）
  ↓ [监管方发布检查计划后自动创建并下达]
assigned（已下达）
  ↓ [服务方确认接收]
accepted（已接收）
  ↓ [服务方开始现场检查]
inspecting（检查中）
  ↓ [完成全部检查项录入]
report_drafting（报告编制中）
  ↓ [提交检查报告]
report_submitted（报告已提交）
  ↓ [监管方审核报告]
  ├→ report_rejected（报告退回）→ 返回 report_drafting
  ↓
under_review（审核中）
  ↓ [审核通过，等待整改完成]
rectifying（整改中）
  ↓ [全部隐患整改完成且复查通过]
completed（已完成）

异常状态：
- overdue（已逾期）：超过计划检查日期未开始
- cancelled（已取消）：监管方取消任务
```

**页面说明：**

- 列表页：任务列表，支持按状态/时间/企业/检查人员筛选

### 2.4.6 检查报告管理

主要用户为服务方及履行方，也是两者默认开启的功能。

**检查报告实体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string(UUID) | 是 | 报告唯一标识 |
| teamId | string(FK) | 是 | 所属团队 |
| taskId | string(FK) | 是 | 关联检查任务 |
| reportNo | string | 是 | 报告编号（自动生成，格式：BG-2025-001） |
| generatedBy | string(FK) | 是 | 报告生成人 |
| generatedAt | datetime | 是 | 报告生成时间 |
| status | enum | 是 | 报告状态：draft / submitted / approved / rejected |
| shareLink | string | 否 | 分享链接（生成后自动创建） |
| version | number | 是 | 报告版本号（退回后重新提交版本+1） |

**报告标准章节结构：**

1. **封面**：报告编号、被检查企业名称、检查日期、检查机构名称、报告生成日期
2. **检查概述**：检查计划名称、检查类型、检查时间（起止）、检查人员列表（姓名/资质编号）、检查范围
3. **企业基本情况**：企业名称、统一社会信用代码、法定代表人、安全负责人、所属行业、企业地址、员工人数
4. **检查依据**：适用的法律法规、标准规范列表
5. **检查发现（隐患问题清单）**：以表格形式列出所有隐患，每条含序号、隐患类别、隐患描述、隐患位置、隐患等级、违反条款、现场照片、整改建议、整改期限、整改责任人
6. **整改情况汇总**：已整改数、未整改数、整改率、各隐患的整改状态
7. **综合评价**：合格率、安全等级评定（A/B/C/D）、总体评价意见
8. **附件**：现场照片汇总、依据文件

**页面说明：**

- 列表页：报告列表，支持按状态/企业/时间筛选
- 在线预览页：按报告章节结构渲染的在线查看页面
- 操作：在线查看、链接分享（生成可分享的临时链接）、导出为PDF、导出为Word

- 数据密度：信息列表适度紧凑，每行高度48px，适合大量数据浏览
- 状态可视化：所有业务状态使用统一的颜色标签系统

# 6. AI开发工作说明

## 6.1 原型及方案探索

- 开发前首先做好安全生产类产品的行业调研并充分了解三方安全检查管理的应用产品形态
- 输出描述完整功能页面的原型或Demo，询问发起者方案及原型审核无误后进行代码开发层面的工作
- 平台总体样式参考常见的国内安全管理平台的导航及功能交互方式，以列表→详情→弹窗的使用习惯保持平台整体交互效果一致
- 文中所列字段信息有限，可收集调研类似行业平台相关字段信息进行增加
- 所有列表支持排序、筛选、字段显示/隐藏、查询、下载、导入、导出功能，可以先放按钮后续考虑单独实现

## 6.2 技术开发实现

- 参考规则引入必要的工具搭建必要的开发环境
- 先分析各功能所需字段关系及功能关系再进行分块功能开发以避免模块间不一致或重复开发的情况
- 符合可上线产品的完整能力及稳定性，如有本文未明确的资源需求或技术需求立即让我补充

## 6.3 开发优先级建议（新增）

建议按以下阶段递进开发：

**第一阶段（核心框架）：** 用户注册登录、团队创建与加入、基础布局与导航框架、角色权限基础

**第二阶段（核心业务）：** 检查计划CRUD、检查任务全流程、隐患录入与管理、隐患整改流程

**第三阶段（输出与分析）：** 检查报告生成与预览、报告导出（PDF/Word）、三方Dashboard数据看板、数据统计分析

**第四阶段（完善体验）：** 通知中心、组织架构管理、列表高级功能（导入/导出/批量）、企业信息管理
